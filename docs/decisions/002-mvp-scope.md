# ADR-002: MVP 범위를 P0 / P1 / Later로 분리

> 상태: 확정
> 작성일: 2026-04-04

---

## 맥락

해다 MVP는 병원 파일럿(4주, 50명 이하)에서 핵심 루프를 검증하는 것이 목표다. 초기 PRD v0.1에서는 모든 기능이 "MVP"로 묶여 있어, 어디까지 만들어야 첫 릴리스가 가능한지 경계가 불분명했다.

---

## 결정

기능을 **P0 / P1 / Later** 세 단계로 분리한다.

---

## P0 — MVP 첫 릴리스

파일럿에서 핵심 루프(로그인 → 챌린지 참여 → 인증 → 달력 확인 → 댓글 → 완료)를 검증하기 위한 최소 범위.

| ID | 기능 | 근거 |
|----|------|------|
| F-01 | 카카오 소셜 로그인 | 앱 진입의 전제 조건 |
| F-02 | 프로필 설정 | 닉네임·사진이 달력 썸네일과 댓글에 필요 |
| F-03 | 챌린지 생성 | 운영자가 챌린지를 만들어야 파일럿 시작 가능 |
| F-06 | 챌린지 참여 (초대 링크) | 운영자가 링크를 공유하여 참여자 모집 |
| F-07 | 챌린지 완료 | 파일럿 기간 종료 시 결과 확인, 동기부여 완결 |
| F-08 | 내 챌린지 목록 (내 페이지) | 앱 메인 화면, 챌린지 공간 진입점 |
| F-09 | 달력 기반 협력 뷰 | 제품 핵심 가치: "함께 채우는 달력" |
| F-10 | 개인 인증 표시 (썸네일) | 달력에서 누가 인증했는지 확인 |
| F-11 | 전원 인증 계절 아이콘 | 제품 핵심 가치: 전원 인증 보상 |
| F-12 | 사람별 인증 내역 조회 | 달력 → 날짜 → 사람 → 상세 탐색 |
| F-13 | 인증 제출 | 핵심 루프의 중심 행위 |
| F-14 | 참여자 간 댓글 | 상호 격려, "으쌰으쌰" 가치 검증 |
| F-17 | 사용 데이터 수집 | 파일럿 성공 지표 측정에 필수 |

### P0에서 의도적으로 단순화한 사항

| 항목 | P0 동작 | 이유 |
|------|---------|------|
| `is_public` | 서버에서 항상 `false` 설정 | 탐색 UI 없이 초대 링크로만 참여 |
| 초대 코드 UI | 생성 완료 화면에서 링크 복사만 제공 | 별도 공유 UI는 P1 |
| 하단 탭 | 내 페이지만 (탐색·알림 탭 없음) | P1 기능이 추가될 때 탭 확장 |
| 푸시 알림 | 미구현 | 파일럿 규모에서 앱 자체 리마인더로 충분 |

## P1 — MVP 이후 확장

P0 파일럿 결과를 확인한 뒤 순차적으로 추가한다.

| ID | 기능 | 전제 조건 | 구현 난이도 |
|----|------|-----------|------------|
| F-04 | 초대 코드 전용 화면 | — | 낮음 |
| F-05 | 공개 챌린지 탐색 | 탐색 탭 UI, `is_public` 활성화 | 중간 |
| F-15 | 인증 리마인더 (푸시) | F-18 (디바이스 토큰) | 중간 |
| F-16 | 팀원 인증 알림 (푸시) | F-18 (디바이스 토큰) | 중간 |
| F-18 | 디바이스 토큰 등록 | FCM 연동 | 중간 |
| F-19 | 알림 목록 화면 | F-15/F-16 | 낮음 |

### P1 구현 순서 (권장)

```
F-18 디바이스 토큰 → F-15 리마인더 → F-16 팀원 알림 → F-19 알림 목록
                                                           │
F-04 초대 코드 화면 (독립)                                   │
F-05 공개 탐색 (독립)                                        │
                                                           ▼
                                                     하단 탭 확장
                                                (내 페이지 | 탐색 | 알림)
```

## Later — MVP 이후 별도 기획

파일럿과 P1 결과를 보고 기획 여부를 결정한다. 현재 코드·스키마에 반영하지 않는다.

| 기능 | 비고 |
|------|------|
| 관리자 대시보드 | 파일럿에서는 서버 로그 + 수기 분석으로 대체 |
| 달성률 기반 랭킹/등급 | 경쟁보다 협력에 집중하는 제품 방향. 파일럿 피드백 후 재검토 |
| 앱 내 설문 / 피드백 폼 | 운영자 수기 + 종료 인터뷰로 대체 |
| 채팅 / 실시간 메시지 | 댓글로 충분. WebSocket 인프라 복잡도 높음 |
| 챌린지 템플릿 / 추천 | 사용 패턴이 쌓인 후에 의미 있음 |
| Apple 로그인 | iOS 앱스토어 심사 시 추가 필요 |
| 챌린지 삭제 / 탈퇴 | 데이터 정합성 정책 확정 후 (Open Question) |

---

## 이 결정의 근거

1. **파일럿 목표에 집중**: 핵심 루프만 빠르게 만들어 "함께 채우는 달력"의 동기부여 효과를 검증한다.
2. **불확실성 축소**: 공개 탐색, 푸시 알림 등은 사용자 반응을 보고 우선순위를 조정할 수 있다.
3. **개발 속도**: P0 범위를 좁혀서 첫 릴리스까지의 개발 기간을 단축한다.
4. **운영 단순화**: 파일럿에서 운영자가 챌린지를 생성하고 링크를 공유하는 시나리오이므로, 탐색·알림 없이도 핵심 루프가 동작한다.
