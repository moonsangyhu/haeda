# ADR-001: 핵심 객체를 Challenge로 결정

> 상태: 확정
> 작성일: 2026-04-04

---

## 맥락

해다 앱에서 사용자 활동(인증, 댓글, 달력, 완료)이 어떤 단위로 묶이는지를 결정해야 한다. 후보는 두 가지였다:

1. **Category(카테고리) 중심**: 카테고리가 독립 엔터티이고, 그 안에 여러 챌린지가 속하는 구조
2. **Challenge(챌린지) 중심**: 챌린지가 최상위 단위이고, 카테고리는 챌린지의 자유 입력 속성

---

## 결정

**Challenge(챌린지)를 핵심 객체로 채택한다.**

- category는 Challenge 테이블의 `VARCHAR(50)` 자유 입력 필드로 유지한다.
- 독립 Category 엔터티(테이블)는 만들지 않는다.
- 앱 화면, API 경로, 라우팅 모두 챌린지 ID를 최상위 그룹으로 사용한다.

---

## 비교

### 선택: Challenge 중심

| 항목 | 내용 |
|------|------|
| 데이터 모델 | Challenge 테이블에 `category VARCHAR(50)` 속성 |
| API 경로 | `/challenges/{id}/calendar`, `/challenges/{id}/verifications` |
| 라우팅 | `/challenges/:id` → 챌린지 공간 |
| 내 페이지 | 챌린지 카드 목록 (카테고리는 카드 안 라벨) |

**장점:**
- 사용자의 실제 활동 단위와 일치: 인증, 댓글, 달력, 완료 모두 챌린지 단위
- API와 라우팅이 직관적: 하나의 ID로 모든 하위 리소스 접근
- 스키마가 단순: 별도 Category 테이블 불필요, JOIN 감소
- 카테고리 자유 입력이 MVP에서 유연함 제공: 사전 정의된 목록 관리 필요 없음
- 내 페이지에서 바로 챌린지로 진입 가능 (1 tap)

**단점:**
- 동일 카테고리 챌린지를 그룹핑하려면 문자열 매칭 필요 (오타, 대소문자 불일치 가능)
- 나중에 카테고리별 추천/탐색을 구현할 때 정규화가 필요할 수 있음

### 포기한 대안: Category 중심

| 항목 | 내용 |
|------|------|
| 데이터 모델 | 독립 Category 테이블 + Challenge.category_id FK |
| API 경로 | `/categories/{id}/challenges/{id}/...` 또는 이중 경로 |
| 라우팅 | `/categories/:catId/challenges/:id` → 중첩 구조 |
| 내 페이지 | 카테고리 폴더 → 챌린지 목록 → 챌린지 공간 (2 tap) |

**이 대안을 포기한 이유:**

1. **사용자 멘탈 모델 불일치**: 사용자는 "운동 30일 챌린지"를 떠올리지, "운동 카테고리"를 먼저 떠올리지 않는다. 챌린지가 활동의 자연스러운 단위다.
2. **탐색 뎁스 증가**: 내 페이지 → 카테고리 → 챌린지 → 달력으로 한 단계가 추가된다. 파일럿 사용자(병원 환자/직원)에게 불필요한 복잡도.
3. **카테고리 관리 오버헤드**: 사전 정의 카테고리를 관리하거나, 사용자 생성 카테고리의 중복/유사어를 처리해야 한다.
4. **API 복잡도**: 경로가 중첩되거나 이중 경로가 필요하고, 카테고리 CRUD API가 추가된다.
5. **MVP 범위 초과**: P0에서 카테고리별 탐색(F-05)은 빠져 있으므로, 카테고리를 1급 엔터티로 만들 실질적 이유가 없다.

---

## 마이그레이션 경로

P1 이후 카테고리별 탐색이 필요해지면:

1. **경량 접근 (권장)**: `Challenge.category` 컬럼에 인덱스 추가, API에서 category 필터링 지원. 엔터티 분리 없이도 충분.
2. **정규화 접근**: Category 테이블 생성, 기존 자유 입력 값 마이그레이션. Challenge에 `category_id` FK 추가. 필요 시점에 결정.
