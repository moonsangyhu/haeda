# 해다 (Haeda) — 제품 요구사항 문서 (PRD)

> 버전: 0.2 (MVP)
> 최종 수정: 2026-04-04

---

## 1. 제품 개요

### 1.1 문제
개인 챌린지 앱은 혼자 하다 보면 동기부여가 부족해 쉽게 포기하게 된다.

### 1.2 솔루션
**참여자 전원이 함께 인증해야 예쁜 계절 아이콘이 완성되는** 협력형 챌린지 앱.
개인이 인증하면 달력에 프로필 썸네일이 표시되고, 해당 날짜에 참여자 전원이 인증을 완료하면 계절감 있는 아이콘(봄: 꽃, 여름: 나뭇잎, 가을: 단풍, 겨울: 눈꽃)으로 바뀐다. "예쁜 달력을 함께 만들자"는 동기가 참여자 간 격려를 자연스럽게 유도한다.

### 1.3 핵심 가치
- **함께하는 으쌰으쌰** — 협력형 인증으로 상호 격려
- **귀엽고 계절감 있는 시각화** — 전원 인증 시 보상이 되는 시각적 즐거움
- **가벼운 일상 기록** — 인증 사진 + 일기 + 댓글로 소통

### 1.4 대상 사용자
- 1차: 병원 파일럿 (환자 + 직원, 대상 미확정)
- 2차: 일반 사용자 확장

### 1.5 핵심 객체
앱의 중심 단위는 **챌린지(Challenge)**이다.
- 사용자는 챌린지를 생성하거나 참여하고, 챌린지 안에서 인증·댓글·달력 활동이 이루어진다.
- **카테고리(category)**는 챌린지의 자유 입력 속성이며, 독립 엔터티가 아니다.
- 내 페이지는 챌린지 목록을 표시하고, 라우팅은 챌린지 ID 기반이다.
- API 경로, 도메인 모델, 화면 모두 챌린지를 최상위 그룹으로 사용한다.

### 1.6 기술 스택
| 레이어 | 기술 |
|--------|------|
| 프론트엔드 | Flutter (iOS / Android) |
| 백엔드 | Python FastAPI |
| 데이터베이스 | PostgreSQL |
| 인증 | 카카오 OAuth |
| 푸시 알림 | FCM (Firebase Cloud Messaging) — P1 |

## 2. 기능 목록 — P0 (MVP 첫 릴리스)

P0은 병원 파일럿에서 핵심 루프(로그인 → 챌린지 참여 → 인증 → 달력 확인 → 댓글 → 완료)를 검증하기 위한 최소 범위이다.

### 2.1 인증/사용자
| ID | 기능 | 설명 |
|----|------|------|
| F-01 | 카카오 소셜 로그인 | 카카오 OAuth를 통한 회원가입/로그인 |
| F-02 | 프로필 설정 | 닉네임, 프로필 사진 설정 |

### 2.2 챌린지
| ID | 기능 | 설명 |
|----|------|------|
| F-03 | 챌린지 생성 | 제목, 설명, 카테고리(자유 입력), 기간(시작일~종료일), 인증 빈도(매일/주N회), 사진 필수 여부 설정 |
| F-06 | 챌린지 참여 | 생성자가 공유한 링크/코드를 통한 직접 참여 (인원 제한 없음) |
| F-07 | 챌린지 완료 | 기간 종료 시 자동 완료, 개인별 달성률 표시, 완료 배지 부여 |

### 2.3 내 페이지
| ID | 기능 | 설명 |
|----|------|------|
| F-08 | 내 챌린지 목록 | 참여 중인 챌린지 카드 목록 + 각 챌린지 달성률 요약 |

### 2.4 챌린지 공간 (달력 뷰)
| ID | 기능 | 설명 |
|----|------|------|
| F-09 | 달력 기반 협력 뷰 | 월별 달력에 날짜별 인증 현황 표시 |
| F-10 | 개인 인증 표시 | 인증한 사람의 프로필 사진 썸네일이 해당 날짜에 표시 |
| F-11 | 전원 인증 계절 아이콘 | 해당 날짜에 참여자 전원이 인증하면 계절 아이콘으로 변환 |
| F-12 | 사람별 인증 내역 조회 | 달력에서 날짜 → 사람 선택 시 인증 상세(사진+일기) 조회 |

### 2.5 인증 & 피드백
| ID | 기능 | 설명 |
|----|------|------|
| F-13 | 인증 제출 | 사진(필수/선택, 챌린지 설정에 따름) + 일기(텍스트) 제출. 사용자가 "하루 경계 시각"을 0/1/2시 중 설정할 수 있어, 새벽(자기 전) 인증을 전날 미션으로 인정받는 것이 가능 |
| F-14 | 참여자 간 댓글 | 인증 내역에 대한 댓글/응원 작성 |

### 2.6 챌린지 완료 경험
| ID | 기능 | 설명 |
|----|------|------|
| F-07 | 챌린지 완료 화면 | 나의 달성률, 완료 배지, 참여자별 달성률, 보존된 달력 표시 |

### 2.7 파일럿 피드백
| ID | 기능 | 설명 |
|----|------|------|
| F-17 | 사용 데이터 수집 | 접속 빈도, 인증율, 댓글 수 등 핵심 지표 트래킹 |

## 3. 기능 목록 — P1 (MVP 이후 확장)

P0 파일럿 결과를 확인한 뒤 순차적으로 추가한다.

| ID | 기능 | 설명 | 비고 |
|----|------|------|------|
| F-04 | 초대 코드 화면 | 챌린지 생성 시 초대 코드 자동 생성 + 복사/공유 UI | P0에서는 서버에서 코드 생성만 하고, 공유는 링크 복사로 대체 |
| F-05 | 공개 챌린지 탐색 | 공개 챌린지 목록 조회 및 참여, 카테고리 필터 | 탐색 탭 |
| F-15 | 인증 리마인더 | 아직 인증하지 않은 날에 리마인더 푸시 | FCM 연동 필요 |
| F-16 | 팀원 인증 알림 | 같은 챌린지 참여자가 인증했을 때 알림 | FCM 연동 필요 |
| F-18 | 디바이스 토큰 등록 | FCM 토큰 등록 및 관리 | 알림의 전제 조건 |
| F-19 | 알림 목록 화면 | 앱 내 알림 히스토리 조회 | 알림 탭 |

## 4. MVP 제외 범위

| 기능 | 제외 사유 |
|------|-----------|
| 관리자 대시보드 | MVP는 사용자 중심, 관리 기능은 이후 |

| 챌린지 템플릿/추천 | 사용자 자유 생성으로 시작 |
| Apple 로그인 | iOS 심사 시 추가 예정 |

## 5. 성공 지표

병원 파일럿 기간(4주) 동안 아래 지표를 측정한다.

| 지표 | 목표 | 측정 방법 |
|------|------|-----------|
| 챌린지 생성 수 | 파일럿 그룹당 1개 이상 | 서버 로그 |
| 일간 인증률 (DAU 중 인증 제출 비율) | 60% 이상 | 서버 로그 (`Verification` 생성 수 / 활성 멤버 수) |
| 전원 인증 달성 일수 비율 | 30% 이상 | `DayCompletion` 레코드 수 / 챌린지 기간 일수 |
| 댓글 작성률 (인증 건당 평균 댓글 수) | 0.5건 이상 | `Comment` 수 / `Verification` 수 |
| 7일 리텐션 (가입 후 7일 뒤 재접속 비율) | 50% 이상 | 서버 접속 로그 |
| 챌린지 완료율 (종료까지 참여 유지 비율) | 70% 이상 | 완료 시점 활성 멤버 / 최초 참여 멤버 |

## 6. 비기능 요구사항

| 항목 | 요구사항 |
|------|----------|
| 응답 시간 | API 95th percentile ≤ 500ms |
| 동시 사용자 | 파일럿 기준 50명 이하 — 수평 확장은 P1 이후 |
| 가용성 | 파일럿 기간 중 일 평균 가동률 99% 이상 (계획된 점검 제외) |
| 이미지 업로드 | 최대 10MB, JPEG/PNG만 허용, 리사이즈 후 저장 |
| 보안 | HTTPS 필수, 카카오 OAuth 토큰 서버 사이드 검증, 사진 URL은 서명된 URL 사용 |
| 데이터 보존 | 파일럿 종료 후 최소 6개월 보존 (분석 목적) |
| 로깅 | 핵심 이벤트(인증 제출, 챌린지 생성/완료, 로그인) 구조화 로그 수집 |

## 7. 병원 파일럿 운영 가정

| 항목 | 가정 |
|------|------|
| 대상 | 1개 병원, 환자 + 직원 혼합 그룹 (정확한 구성 미확정 → Open Questions) |
| 기간 | 4주 |
| 그룹 규모 | 챌린지당 5~15명 |
| 운영자 | 병원 담당자 1명 + 해다 팀 1명 |
| 디바이스 | 참여자 개인 스마트폰 (BYOD), Android 비율이 높을 것으로 예상 |
| 네트워크 | 병원 Wi-Fi 또는 개인 모바일 데이터 |
| 온보딩 | 운영자가 대면으로 앱 설치 및 카카오 로그인 안내 |
| 챌린지 생성 | 운영자가 대신 생성 후 참여자에게 초대 링크 공유 |

## 8. 피드백 수집 방식

파일럿 기간 동안 아래 세 가지 채널로 피드백을 수집한다.

| 채널 | 방법 | 수집 주기 | 담당 |
|------|------|-----------|------|
| 사용 로그 | 서버 구조화 로그에서 핵심 지표 추출 (§5 성공 지표) | 실시간 수집, 주간 리포트 | 해다 팀 |
| 운영자 수기 수집 | 병원 운영자가 참여자 반응·불편 사항을 메신저/스프레드시트로 전달 | 주 1회 | 병원 담당자 |
| 파일럿 종료 인터뷰 | 참여자 5명 이상 대면/비대면 인터뷰 | 종료 후 1회 | 해다 팀 |

> 앱 내 설문 기능은 MVP에 포함하지 않는다. 사용 로그 + 운영자 수기 + 종료 인터뷰로 충분히 검증한 뒤, 필요 시 P1에서 앱 내 피드백 기능을 추가한다.
